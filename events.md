# Event Catalog

Project: Chores Cooperative (Chocoop)  
Status: Active  
Last Updated: 2026-05-02  
Owner: Sebastian Celejewski 

# Overview

This document defines domain events used in the ChoCoop platform.

Events represent facts that already happened and are used for asynchronous communication between components.

All events follow conventions defined in ADR-002.

# Common Event Fields

All events contain the following fields:

- eventName — name of the event  
- occurredAt — ISO timestamp of when the event occurred  
- source — originating system (e.g. sebcel-chocoop-app)  

# Identifiers

- all entity identifiers are GUIDs  
- userId is the Cognito `sub`  
- identifiers are opaque and carry no business meaning  

# Events

## WorkRequestCreated

Description:  
Emitted when a new work request is created.

Fields:

- workRequestId (GUID)  
- userId (GUID, Cognito sub)  
- type (string)  
- exp (number)  

Notes:

- type corresponds to activity category  

## WorkRequestCompleted

Description:  
Emitted when a work request is completed and converted into an activity.

Fields:

- workRequestId (GUID)  
- activityId (GUID)  
- userId (GUID)  

Notes:

- activityId references the created Activity  

## ActivityCreated

Description:  
Emitted when a new activity is recorded.

Fields:

- activityId (GUID)  
- userId (GUID)  
- type (string)  
- exp (number)  
- date (ISO date)  

Notes:

- date represents when the activity was performed  

## WorkRequestOverdue (planned)

Description:  
Emitted when a work request becomes overdue based on time evaluation.

Fields:

- workRequestId (GUID)  
- detectedAt (timestamp)  

Notes:

- generated by scheduler / time-based processing  

## KimbalontekMoodChanged (planned)

Description:  
Emitted when Kimbalontek mood changes.

Fields:

- mood (string: happy | neutral | sad)  
- previousMood (string)  
- changedAt (timestamp)  

Notes:

- generated by Kimbalontek engine  
- used for notifications and UI updates  

# Evolution Rules

- events are immutable  
- existing fields must not be removed  
- existing field semantics must not change  
- new fields must be optional  
- consumers must ignore unknown fields  

Breaking changes require:

- a new event type, or  
- explicit versioning
